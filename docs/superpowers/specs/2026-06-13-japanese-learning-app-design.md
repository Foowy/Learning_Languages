# Japanese Learning App — Design Spec

**Date:** 2026-06-13  
**Status:** Approved  

---

## Overview

A personal Japanese learning application for a complete beginner. Covers both written (hiragana, katakana, kanji) and spoken (listening, pronunciation) skills through structured, step-by-step lessons that feed into spaced-repetition review sessions. Runs as a Docker container on a home Linux server, accessible from desktop and tablet via a browser.

---

## Goals

- Learn to read and write Japanese scripts (hiragana → katakana → kanji)
- Build speaking and listening skills from day one
- Structured lesson progression with daily SRS review
- Accessible from any device on the home network (desktop, tablet)
- Single user, no authentication required
- Everything runs locally — no cloud dependency except edge-tts for TTS audio

---

## Architecture

### Deployment

The app runs as a Docker container on a home Linux server. The user's existing Caddy instance handles HTTPS termination and reverse proxies to the container.

```
Desktop / Tablet (browser)
        ↕ HTTPS (Caddy → http://app:13200)
  Linux Server — Docker Container
        ↕ volume mounts
  Named Docker Volume (SQLite + audio cache + lesson JSON)
```

### Components

| Component | Technology |
|---|---|
| Backend | Python 3.11, FastAPI, uvicorn |
| Frontend | Vanilla HTML/CSS/JS, served by FastAPI (no build step) |
| Database | SQLite via `aiosqlite` |
| TTS | `edge-tts` (Microsoft neural TTS, Nanami-san JP voice) |
| STT | `openai-whisper` (local, `base` model, ~140MB) |
| Container | Docker + Docker Compose |
| HTTPS | Caddy (existing, user-managed) |

FastAPI binds to `0.0.0.0:13200`. The container exposes port 13200. Caddy proxies HTTPS traffic to it — no additional cert setup needed inside the app.

### Data Storage

All data lives in a named Docker volume mounted at `/data`:

- `/data/progress.db` — SQLite database (progress, SRS state, lesson completions)
- `/data/audio/` — cached MP3 files generated by edge-tts
- `/data/lessons/` — JSON lesson content files (bundled with app, copied to volume on first run)

## Curriculum

Five sequential units. Each lesson introduces ~5 characters or concepts.

| Unit | Content | Lessons |
|---|---|---|
| 1 | Hiragana (46 characters, grouped by row) | ~10 |
| 2 | Katakana (46 characters, grouped by row) | ~10 |
| 3 | Basic vocabulary (N5 words using learned scripts) | ~20 |
| 4 | Basic kanji (N5 kanji, ~100 characters) | ~20 |
| 5 | Phrases and conversation | Ongoing |

Units unlock sequentially. Within each unit, lessons unlock as the previous lesson is completed.

---

## Lesson Flow

Each lesson has four phases, always in this order:

### Phase 1 — Introduce
- Display each character with its romanization and meaning
- Play native audio (edge-tts) for each character
- Show animated stroke order (SVG-based)
- Show an example word that uses the character

### Phase 2 — Listen & Speak
- Play a character or short sequence via TTS
- User records themselves via browser mic (Web Audio API)
- Audio POST'd to `/api/speech/recognize` → Whisper transcribes
- Frontend shows "Whisper heard: X" and highlights match/mismatch
- No accent scoring — transcription match is the feedback signal

### Phase 3 — Write
- HTML5 Canvas drawing area with stroke order guide overlay
- User traces the character, then draws from memory
- Touch/stylus supported (tablet-friendly)
- Also supports romaji-to-kana typing as an alternative

### Phase 4 — Mini Quiz
- 3–5 questions mixing: character → sound, audio → character, type what you hear
- Score recorded to `lessons` table
- Missed characters queued for SRS review

After completing all four phases, the lesson is marked complete and the next lesson unlocks.

---

## SRS Review

Daily review sessions surface cards that are due based on the SM-2 algorithm.

- Each card has: `interval_days`, `ease_factor`, `due_date`, `review_count`
- After each review, the user rates recall (1–4) and SM-2 updates the schedule
- New cards enter review after first lesson completion
- Review sessions are separate from lessons (accessible from the dashboard)

## UI Layout

**Top navigation bar** with links: Home · Lessons · Review · Speak · (Settings gear).

**Dashboard (Home):**
- Welcome message + streak count
- Primary CTA: "Continue Lesson N" button
- Secondary CTA: "Review — X cards due" button
- Progress bars per unit (hiragana, katakana, etc.)
- Recently learned characters displayed as a tile grid

**Lesson view:** Full-width, phase-by-phase flow. Back/skip navigation between phases. Progress indicator showing current phase.

**Review view:** Flashcard-style. Show front (character or audio), user responds, flip to reveal, rate recall.

**Speak view:** Freeform speaking practice — pick a character set, hear it, speak back, see transcription. Separate from structured lesson speaking phases.

**Responsive design:** Layout adapts to tablet screen sizes. Top nav collapses to hamburger menu on small screens. Canvas writing area is touch/stylus-friendly.

## Speech Pipeline

### TTS (Text → Audio)
1. Backend receives text to speak
2. Check audio cache (`/data/audio/<hash>.mp3`)
3. If not cached: call `edge-tts` with `ja-JP-NanamiNeural` voice, save to cache
4. Stream MP3 to browser

### STT (Mic → Text)
1. Browser records mic audio via Web Audio API, encodes as WebM/WAV
2. POST audio blob to `/api/speech/recognize`
3. Backend saves temp file, runs `whisper.transcribe()` with `base` model, language=`ja`
4. Return transcription JSON to frontend
5. Frontend compares to expected text, highlights result

Whisper `base` model runs on CPU, typically 2–5 seconds for short inputs (single characters, short words). No GPU required.

## Data Model

### `cards` table
Seeded from lesson JSON on first run.

| Column | Type | Description |
|---|---|---|
| id | INTEGER PK | |
| type | TEXT | `hiragana`, `katakana`, `kanji`, `vocab` |
| character | TEXT | The Japanese character/word |
| romaji | TEXT | Romanization |
| meaning | TEXT | English meaning |
| unit | INTEGER | Curriculum unit number |
| lesson | INTEGER | Lesson number within unit |
| audio_path | TEXT | Cached TTS file path (nullable) |

### `progress` table
One row per card, created when card is first reviewed.

| Column | Type | Description |
|---|---|---|
| card_id | INTEGER FK | |
| due_date | DATE | Next review date |
| interval_days | INTEGER | Current SRS interval |
| ease_factor | REAL | SM-2 ease factor (default 2.5) |
| review_count | INTEGER | Total reviews |
| last_score | INTEGER | Last recall rating (1–4) |

### `lessons` table

| Column | Type | Description |
|---|---|---|
| lesson_id | INTEGER | Lesson identifier |
| unit | INTEGER | Unit number |
| completed_at | DATETIME | When lesson was finished |
| quiz_score | INTEGER | Phase 4 score (correct/total) |

## Docker Compose

```yaml
services:
  app:
    build: .
    ports:
      - "13200:13200"
    volumes:
      - japanese_data:/data
    restart: unless-stopped
    environment:
      - DATA_DIR=/data
      - PORT=13200

volumes:
  japanese_data:
```

Single `docker compose up -d` to deploy. Data persists across restarts and image updates.

## Error Handling

- **edge-tts unavailable (no internet):** Fallback to cached audio if available; show silent placeholder if not. Lessons remain functional without audio.
- **Whisper transcription timeout:** 10-second timeout on STT endpoint; return error JSON, frontend shows "Try again" without blocking lesson progress.
- **Canvas not supported:** Graceful degradation to romaji typing for writing phase.
- **Mic access denied:** Speaking phase skips gracefully with a "mic unavailable" notice; lesson can still be completed.

---

## Out of Scope

- Multiple user accounts
- Cloud sync or backup (user manages Docker volume backups)
- Mobile native app (browser-based is sufficient)
- Pronunciation scoring beyond transcription match
- Grammar lessons (vocabulary and scripts only in initial scope)
